# ESP-111 项目与开发长期硬约束

## 1. 项目架构与职责

- ESPC51、ESPC52 必须作为 ESP32-C5 房间终端；ESPS3 必须作为 ESP32-S3 本地网关；ESP-server 必须作为后端事实存储和业务服务。
- C5 仅允许访问 ESPS3 暴露的 `/local/v1/*`。禁止 C5 直接访问公网、ESP-server、`/api/*` 或 `/kernel/*`。
- 所有 Server-facing 请求仅允许由 ESPS3 发起或代理。C5 不得绕过 S3。
- ESPS3 必须保持 SoftAP 本地链路与 STA 上行链路相互隔离。Server 或 STA 不可用时，C5 到 S3 的本地链路仍必须继续工作。

## 2. C51/C52 一致性

- C51 和 C52 的公共业务逻辑必须保持一致；仅允许设备身份、local id、alias、room 和 CSI link identity 等必要差异。
- 公共修复必须同步修改并检查两端，禁止只修改一个 C5 后造成静默漂移。
- C51 对应 `sensair_shuttle_01`、local id `1`；C52 对应 `sensair_shuttle_02`、local id `2`。

## 3. CSI 边界

- 原始 CSI、I/Q、phase、完整 amplitude 和子载波数组不得离开 C5；C5 仅允许向 S3 上传低维 CSI 特征和状态摘要。
- CSI callback 内禁止 HTTP、复杂算法、大块动态分配和长时间阻塞；CSI 处理必须在 worker 内完成。
- CSI 高频事件必须采用 latest/coalesce 语义，不得逐帧积压。每个 C5 必须独立判断本房间状态；C51 和 C52 位于不同房间，不得融合成无法区分来源的单一房间结果。
- S3 可以进行系统级汇总，但必须保留设备、房间和链路身份。
- 信道、BSSID、CSI 配置或链路上下文变化时必须重新校准。校准期间输出必须为 `calibrating` 或 `unknown`，不得伪造 `idle`、`vacant`。
- CSI 属于可降级业务，必须让位于语音、网络恢复、心跳和命令链路。默认优先轻量自适应规则，未经充分理由不得引入大型模型或重依赖。

## 4. 语音链路约束

- 唤醒前仅保留最低限度的 wake/VAD 能力，不得无必要长期占用大块 internal RAM、DMA RAM、任务栈、HTTP 连接或播放资源。
- 大块普通音频缓存必须优先使用 PSRAM；DMA 必需对象必须保留在 DMA-capable internal RAM。
- 唤醒后必须进入语音独占状态，暂停 CSI、BME、普通命令轮询和低优先级后台任务，同时保留必要的链路存活、中止和错误恢复机制。
- 语音时序必须为：检测唤醒词；播放唤醒提示；提示结束后开启正式录音；用户讲话期间保持 Mic 开启；VAD 判断结束；关闭录音并上传；等待和播放回复；清理本轮资源；恢复后台业务。
- speaker 与正式录音不得同时工作。每轮语音必须使用明确的 state、owner 和 generation；旧轮次 callback、HTTP 响应或清理动作不得影响新轮次。
- 所有失败、中止、断网和超时路径最终必须释放资源并恢复可唤醒状态。不得破坏现有 ASR、LLM、TTS、PCM 格式和上游网关兼容性。

## 5. S3 Wi-Fi 和网络约束

- S3 必须保持 APSTA 模式：SoftAP 服务 C51/C52，STA 负责外部网络和 Server 上行。
- Wi-Fi callback 仅允许更新状态和投递事件；扫描、选网和连接必须由 network worker 串行执行。
- STA 断开不得导致 SoftAP 或本地 C5 session 被错误释放；Server 不可用不得阻止本地 register、heartbeat、status、BME 和 CSI 接入。
- 网络恢复必须具有退避、稳定窗口和 generation。信道或 BSSID 变化必须通知 CSI 链路。
- 状态类数据必须采用 latest/coalesce；可靠送达的数据必须具备独立的重试或确认语义。网络恢复后不得回放大量过期 CSI 高频历史。

## 6. 资源、任务和队列约束

- 禁止通过盲目增加任务栈掩盖资源生命周期或所有权问题。必须区分 internal RAM、DMA-capable RAM、PSRAM、最大连续块和任务栈余量；大块普通数据优先进入 PSRAM。
- callback、ISR 和高优先级任务不得阻塞等待普通队列；核心协议 worker 不得因无限等待阻塞 register、heartbeat、sensor 或 ACK。
- 每个队列必须定义 owner、容量、满队列行为、drop/coalesce/retry 语义和 generation 失效规则。状态类数据允许合并。
- 命令 ACK、控制结果和必须可靠送达的事实不得按普通状态数据静默丢弃。日志必须能够定位 task、queue、stage、owner、generation 和具体错误。

## 7. 命令和状态语义

- C5 收到本地 S3 ACK 不等于 Server 已持久化；S3 接收入队成功不得描述为 Server 提交成功。
- 命令执行必须考虑幂等、重复下发、ACK 丢失和重试。没有真实智能家居设备时必须返回失败或不可用，禁止伪造 `success`、`applied` 或真实设备状态。
- child registry 必须作为 C5 在线状态的主要真源。`voice busy`、`link lost`、`offline` 和 `registered` 必须具有明确且独立的语义。

## 8. Server 和前端保护边界

- 默认不得修改 `ESP-server/public/`，不得修改真实 `ESP-server/db/database.db`。
- 不得为了固件方便随意修改既有 Server API。接口不足时，必须先提出最小合同和计划，不得直接扩大修改范围。
- 不得启动、部署或重启生产 Server，未经授权不得执行真实外部服务 canary。必须保持现有上游网关和语音请求兼容。
- 不得在源码、文档、日志或输出中暴露真实凭据、Token、密码或敏感地址。

## 9. 工作区和 Git 保护

- 当前工作区可能包含未提交改动；修改前必须检查 live source 和 Git 状态，并分别检查顶层仓库与 `ESP-server` 嵌套仓库。
- 禁止覆盖、删除或回退与当前任务无关的改动。禁止执行 `git reset --hard`、`git checkout .`、`git clean`；未经用户明确要求，不得执行 pull、rebase、merge、commit 或 push。
- `build/`、`managed_components/` 和生成文件不得作为手工源码修改目标。旧文档仅作历史参考，当前源码是事实依据。

## 10. 修改原则

- 必须优先最小闭环修改，不得进行大规模重写。先修确定性逻辑、状态机、资源 owner 和失败恢复，再做结构拆分。
- 不得为了重构改变外部协议、失败语义或主链路时序；不得影响 register、heartbeat、status、BME、CSI、command 和 voice 主链。
- 每项新增逻辑必须明确输入、输出、状态、owner、generation、超时、失败恢复和日志。禁止无界队列、无界缓存、无限重试和无限等待。
- 范围外问题仅记录，不得顺手扩大任务。计划与源码冲突时必须以源码为准，并更新相关文档。

## 11. 新功能独立开发与接入约束

### 11.1 独立开发原则

- 所有新功能默认必须以独立模块、独立 component、独立文件或独立测试入口实现；开发和验证阶段不得直接修改已有稳定链路。
- 新功能不得为接入而重写、替换或大范围调整 Wi-Fi、C5 到 S3 本地通信、S3 到 Server 上云、register、heartbeat、status、BME、CSI 正式、command、voice、speaker、Mic、VAD、wake、Dashboard 数据链路。
- 新功能必须先形成可独立编译、静态验证或离线测试的功能孤岛。未经用户明确同意接入主链路前，不得改变现有运行时行为。

### 11.2 默认关闭和启动链保护

- 每个新功能必须有独立开关，且默认关闭；默认构建和默认启动路径必须保持原有行为。
- 功能关闭时不得创建任务或队列、注册 callback、申请大块内存、初始化外设、发送网络请求、改变协议输出或增加主链路时延。禁止“关闭但仍提前初始化资源”的伪关闭。
- 在独立开发阶段，未经用户明确批准，不得修改或接入 `app_main`、C5 `app_orchestrator`、S3 `gateway_orchestrator`、C5/S3 scheduler 或 runtime/network worker 默认启动流程、Wi-Fi callback、CSI callback、voice state machine、BME 周期任务、command polling、register/heartbeat/status 周期、Server 上传队列、Dashboard snapshot，或正式协议公共头既有字段语义。
- 可以新增接口声明、独立模块和测试入口，但不得使其默认进入正式运行路径。

### 11.3 既有合同和允许的扩展方式

- 新功能不得改变既有链路的路由、请求或响应格式、字段含义、错误码、超时、重试策略、优先级、队列行为、资源所有权、状态机转换、上报周期、默认开关及成功或失败语义。
- 确实必须改变已有合同或链路时，必须暂停实施，先说明无法独立完成的原因、必须修改的文件和链路、现有功能影响、兼容方案、回滚方案和验证方案；仅在用户明确授权后继续。
- 新功能必须优先使用新 component、新 `.c/.h` 文件、内部接口、adapter、feature flag、离线测试、mock/sample driver、独立诊断入口、旁路数据结构、既有公开接口或只读快照扩展。
- 禁止直接读取或修改其他模块私有静态状态；禁止通过 `extern` 暴露内部变量；新模块不得反向控制已有模块内部状态。

### 11.4 接口、所有权、失败和资源隔离

- 每个新功能必须明确记录模块职责、输入、输出、public API、私有状态、内存/task/queue/callback owner、生命周期、generation、超时、失败恢复、开启与关闭条件、资源预算、降级能力及对既有链路的影响。
- 跨模块交互必须通过明确接口、事件或只读快照；禁止共享所有权模糊的全局可变状态。
- 新功能初始化失败不得阻止已有系统启动；运行失败、队列满、内存不足、网络失败或算法异常时，只能关闭或降级自身，不得导致已有模块退出、占满公共 worker/event bus/HTTP 资源，或无限等待阻塞已有链路。
- 新功能不得把自身错误伪装为 Wi-Fi、Server、语音或其他模块错误，必须具有独立错误码和日志前缀。
- 实现前必须给出 task 数量、stack、internal RAM、DMA RAM、PSRAM、queue 深度、最大 payload、CPU 周期和网络频率预算。不得盲目增大既有任务栈、挪用既有模块预留资源，或降低语音、Wi-Fi 重连、心跳、命令和核心状态链路的资源优先级。
- 可丢弃的新功能状态数据必须使用 latest/coalesce；禁止无界缓存和历史积压。

### 11.5 分阶段接入

- 阶段 A，独立设计：仅编写接口、数据结构、模块边界和测试计划，不接入正式运行链路。
- 阶段 B，独立实现：在独立文件或 component 内完成逻辑，以离线样本、mock 或显式测试入口验证，默认不开启。
- 阶段 C，旁路验证：仅读取既有链路的只读数据，不改变既有状态机、输出或业务结果。
- 阶段 D，受控接入：必须由用户明确授权，使用默认关闭的 feature flag，先单设备或单路径验证；失败时必须能立即关闭并恢复原链路。
- 阶段 E，正式启用：仅当独立测试、构建、C51/C52 一致性、资源预算、失败隔离、原有链路回归和真机验证均通过，且用户明确同意后，才可考虑默认启用。

### 11.6 接入最小化和文档要求

- 独立模块完成后，如获授权接入而必须修改已有文件，每个已有文件仅允许增加最小连接代码，例如初始化调用、接口注册、feature flag 判断、事件投递或只读状态获取；新功能主体必须保留在独立模块内。
- 禁止借接入机会重构无关代码、顺手修复范围外问题，或混合功能开发与大规模架构重构。
- 每项新功能必须先创建独立计划文档，至少包括目标、非目标、当前链路、独立模块设计、不修改范围、接口、数据结构、状态机、资源预算、失败隔离、feature flag、分阶段接入步骤、回滚方式、验收标准和未验证事项。
- 计划必须明确写入：`本功能在用户批准正式接入前，不修改已有运行链路，不改变默认行为。`

### 11.7 Codex 执行规则

- Codex 收到新功能任务时必须先判断：能否独立实现、是否修改已有链路、是否改变默认行为、是否增加常驻资源、是否改变协议或状态机，以及是否已获得用户接入授权。
- 若任务仅为写计划、实现独立功能或功能验证，Codex 不得自行接入正式主链路。
- 用户未明确要求“接入现有运行链路”“修改正式启动流程”或同等授权时，一律仅允许独立开发、默认关闭、不修改已有链路且不改变现有行为。

## 12. 开发人员实施约束

- 开始修改前必须读取当前源码，确认任务范围和不修改范围，并检查工作区既有改动；不得只依据计划文档，禁止覆盖他人或历史改动。
- 必须优先最小修改；新功能默认独立开发，不得与无关重构混在同一任务。发现范围外问题仅记录，不得擅自修复。
- 不得通过增加任务栈、队列或超时掩盖根因；不得引入无界队列、缓存、重试或等待。
- 不得暴露密码、Token、密钥、内部地址或用户数据；未经用户明确授权，不得修改真实数据库或前端。
- 不得执行破坏性 Git 操作，不得宣称未执行的测试已经通过。计划与源码冲突时必须以源码为准并明确指出冲突。
- 任何突破本文件长期硬约束的修改都必须先获得用户明确授权。

## 13. 验证与报告要求

- 修改后必须进行静态检查；C51/C52 公共逻辑必须进行一致性检查。涉及固件时必须分别验证 C51、C52、S3。
- 构建通过不代表真机、弱网、CSI、语音或端到端验证通过；未执行的验证必须明确标注“未验证”，禁止宣称未实际完成的测试已通过。
- 最终报告至少必须包含：当前真实链路、问题和根因、修改范围、不修改范围、修改文件、状态机变化、资源 owner 和 generation 变化、已执行验证、未执行验证、剩余风险、真机验收建议。

本文件定义 ESP-111 项目及开发过程的长期硬约束。新功能默认必须独立开发、默认关闭，并且不得修改现有稳定链路。只有用户明确授权正式接入或突破边界后，才能修改对应已有链路。
