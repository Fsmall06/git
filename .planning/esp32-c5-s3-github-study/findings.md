# ESP32-C5 / ESP32-S3 GitHub 学习发现

## 需求与假设

- 用户要求持续三小时，广泛学习 GitHub 上 C5、S3 相关开源项目的代码、功能与调度，并写成学习文档。
- 本轮将 C5/S3 解释为 ESP32-C5/ESP32-S3。
- 当前 ESP-111 是 C5 终端 -> S3 网关 -> Server 三层系统；外部研究需最终回映这条边界，但不得直接改业务代码。

## 证据等级

| 等级 | 定义 |
|------|------|
| A | 固定 commit 的源码、构建配置或官方 API 文档直接证明 |
| B | 仓库 README/设计文档与源码结构相互印证 |
| C | issue/discussion/release notes 或代码注释提示，尚未闭环到实现 |
| D | 搜索摘要或二手文章，只可用于发现候选，不进入最终强结论 |

## 项目筛选维度

- 芯片相关性：明确支持 ESP32-C5 或 ESP32-S3，而非泛泛写 ESP32。
- 学习价值：有真实任务、队列、ISR、网络、显示、音频、AI、低功耗或可靠性代码。
- 可证性：能定位到稳定源码路径和 commit。
- 工程成熟度：构建说明、许可证、测试/CI、release 或活跃维护情况清楚。
- 互补性：避免只收集同一类 demo，覆盖官方基础、组件框架与社区产品。

## 研究发现

### 本地基线

- 工作区已有大量用户改动；研究任务不触碰固件、后端、前端和数据库。
- 根 `task_plan.md/findings.md/progress.md` 混有多轮旧审计，本轮采用独立作用域保存上下文。
- 项目规则确认 C5 target 为 `esp32c5`（RISC-V），S3 target 为 `esp32s3`（Xtensa LX7），ESP-IDF 版本为 v5.5.4。

### 第一轮发现方法结论

- GitHub repository search 的 `in:readme` 结果噪声很高：命中只能证明仓库文本提到芯片名，不能证明构建 target 或运行支持。
- 最低验真门槛应是：板级/target 配置、CI build matrix、Kconfig/CMake manifest 或具体源码条件分支至少命中一项；最终“支持”结论优先要求两项相互印证。
- 2026-07-10 查询时 GitHub 未认证 API core 配额为 60 次/小时、search 为 10 次/分钟；后续优先批量 API 和 shallow clone，避免逐文件 API 浪费配额。
- Web 搜索/打开工具连续三次发生同一返回流解码失败，已停用；GitHub REST API 正常，后续来源链接仍直接指向 GitHub 一手仓库/固定 commit。

## 深读记录

每个项目使用以下字段：仓库、commit、许可证、功能、入口、任务拓扑、通信/同步、资源治理、错误恢复、可借鉴点、限制、证据路径。

### `espressif/esp-csi`（深读 1）

- **快照/许可：** `8633d67152db2808f141cc1595970aa9cf406045`，Apache-2.0。
- **芯片证据：** `get-started/csi_recv` 在 C5 分支设置 2.4/5 GHz band/bandwidth，并使用新 `wifi_csi_config_t` 字段；同文件对 S3/旧芯片保留 legacy 配置分支。C5 还显式处理 HE-LTF/12-bit force-LLTF 数据。
- **回调隔离的较好样例：** `esp-radar/console_test/main/app_main.c` 在 CSI 回调中复制一个变长对象，以零等待投递到容量 64 的指针队列；投递失败立即释放。独立 `csi_data_print_task` 阻塞取队列后做重格式化，源码注释明确目标是避免阻塞 CSI 回调。证据：[callback](https://github.com/espressif/esp-csi/blob/8633d67152db2808f141cc1595970aa9cf406045/examples/esp-radar/console_test/main/app_main.c#L115-L125)、[consumer/start](https://github.com/espressif/esp-csi/blob/8633d67152db2808f141cc1595970aa9cf406045/examples/esp-radar/console_test/main/app_main.c#L337-L370)、[queue/task](https://github.com/espressif/esp-csi/blob/8633d67152db2808f141cc1595970aa9cf406045/examples/esp-radar/console_test/main/app_main.c#L730-L739)。
- **较好样例本身的限制：** 回调先 `malloc` 再解引用，未检查分配失败；打印任务优先级为 0，队列深度和堆压力没有指标/水位策略。因此可借鉴“回调只复制+投递、重处理下沉”，不能照搬内存和优先级参数。
- **反例 1：** `esp-crab/master_recv` 的 callback `calloc` 后无空指针检查，`xQueueSend(..., 0)` 返回值被忽略；容量 20 的队列满时分配块不会被 consumer 获得，存在泄漏窗口。consumer 虽会在正常路径释放，但不能覆盖投递失败。证据：[producer](https://github.com/espressif/esp-csi/blob/8633d67152db2808f141cc1595970aa9cf406045/examples/esp-crab/master_recv/main/app_main.c#L111-L152)、[queues/consumer](https://github.com/espressif/esp-csi/blob/8633d67152db2808f141cc1595970aa9cf406045/examples/esp-crab/master_recv/main/app_main.c#L176-L246)。
- **反例 2：** `get-started/csi_recv` 直接在 CSI callback 中循环 `ets_printf` 整帧数据，适合最小 bring-up/观察，不适合生产高频链路。证据：[direct printing callback](https://github.com/espressif/esp-csi/blob/8633d67152db2808f141cc1595970aa9cf406045/examples/get-started/csi_recv/main/app_main.c#L137-L210)。
- **采样与功耗取舍：** `wifi_sensing_demo` 在连接后显式 `esp_wifi_set_ps(WIFI_PS_NONE)`，注释说明为了稳定 CSI cadence；这是测量稳定性优先于省电的项目选择，不是 C5 通用默认。证据：[bring-up](https://github.com/espressif/esp-csi/blob/8633d67152db2808f141cc1595970aa9cf406045/examples/esp-radar/wifi_sensing_demo/main/app_main.c#L200-L215)。
- **网络恢复：** `console_test` 在断连事件内调用 `esp_radar_wifi_reinit()`，而 got-IP 时新建触发任务；没有在该 handler 片段中看到任务句柄去重，需警惕重复 got-IP 造成重复 producer，最终文档只把它列作待核验风险，不作已确认 bug。
- **对 ESP-111 的价值：** 支持当前“C5 callback 只做轻特征/入队、worker 处理”的方向；同时给出必须补齐的生产门槛：分配失败、queue-full 所有权、drop counter/水位、latest-only 或有界背压、重连任务幂等。

### `espressif/esp-idf` 调度与系统基线（深读 2）

- **快照/许可：** `f0887bcf8763266effe3fa0b358340df226a04b5`，Apache-2.0；示例文件常使用 Unlicense/CC0 双许可，复用时仍按具体文件 SPDX 核对。
- **核数是调度结论的前提：** C5 `SOC_CPU_CORES_NUM` 为 1，S3 为 2。证据：[C5 soc_caps](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/soc/esp32c5/include/soc/soc_caps.h#L159-L166)、[S3 soc_caps](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/soc/esp32s3/include/soc/soc_caps.h#L145-L149)。
- **`tskNO_AFFINITY` 不等于 C5 上的隔离：** IDF API 明确说明单核时 `tskNO_AFFINITY` 被当作 core 0；SMP 示例也把第二核在单核 target 上退化为 `tskNO_AFFINITY`。因此 C5 的高优先级 CPU 密集任务会直接挤压所有同核 worker，重点应放在 bounded work/yield/priority，而不是“绑核”。证据：[API](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/freertos/esp_additions/include/freertos/idf_additions.h#L39-L69)、[示例](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/system/freertos/basic_freertos_smp_usage/main/create_task_example.c#L12-L55)。
- **IDF 栈参数单位是 byte：** `xTaskCreatePinnedToCore` 的 stack depth 与 vanilla FreeRTOS 不同，单位为字节。跨项目抄任务参数时若忽略这一点，内存预算会被误读。证据同上 API L49-L53。
- **默认事件循环的隐藏拓扑：** `esp_event_loop_create_default()` 创建容量由 Kconfig 决定的队列和名为 `sys_evt` 的专用任务，固定 `.task_core_id = 0`；C5/S3 都一样。自定义 event loop 才可选择 queue size、priority、core，或不给专用任务由调用者驱动。证据：[default loop](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/esp_event/default_event_loop.c#L92-L108)、[custom loop creation](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/esp_event/esp_event.c#L553-L624)。
- **事件数据与 queue-full 所有权：** 普通 `esp_event_post_to` 会复制 event data；发送失败会删除内部 post 并返回 `ESP_ERR_TIMEOUT`。从 loop 自己的 task 再 post 时强制零等待，避免 self-deadlock；ISR post 只允许装入内联缓冲的小数据并用 FromISR queue API。证据：[post path](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/esp_event/esp_event.c#L960-L1032)、[ISR path](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/esp_event/esp_event.c#L1050-L1080)。调用方仍必须检查返回值，框架只保证不泄漏内部副本，不保证业务事件必达。
- **handler 串行含义：** loop task 从一个队列取事件，在同一 loop 上依次执行 loop/base/id handlers；任何长 handler 都会阻塞该 loop 的后续 Wi-Fi/IP/业务事件。S3 有双核并不改变这个串行合同。
- **Task WDT 可监控“任务”也可监控“任务内阶段”：** 除 `esp_task_wdt_add/reset` 外，API 提供 named user handle，适合在一个长期 worker 内给关键周期函数分别记活性。官方示例同时订阅 task、`func_a`、`func_b`，停止时逐项注销并用 task notification 等待退出。证据：[example](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/system/task_watchdog/main/task_watchdog_example_main.c#L18-L101)、[API contract](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/esp_system/include/esp_task_wdt.h#L79-L157)。
- **队列+通知适合批处理，但 demo 参数不是生产策略：** batch example 用 queue 传数据、task notification 触发五件一批、mutex 保护计数；producer 却 `portMAX_DELAY` 入队，并在锁外读计数。可借鉴“数据和唤醒分离”，不可照搬无限等待和共享计数。证据：[batch example](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/system/freertos/basic_freertos_smp_usage/main/batch_processing_example.c#L10-L121)。
- **S3 PSRAM 的边界：** IDF 允许为 task stack/queue 指定 memory caps，但 TCB 始终在 internal RAM，且这类对象必须用配套 delete API；文档还建议跨任务删除，确保目标未在另一核运行。适合把非实时大栈/大队列放 PSRAM，不代表 ISR/低延迟控制结构也应迁出。证据：[caps task/queue API](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/freertos/esp_additions/include/freertos/idf_additions.h#L256-L383)。
- **对 ESP-111 的直接含义：** C5 dispatcher/worker 必须以单核总预算设计；S3 可以把网络、融合或 UI 类 CPU 工作分核，但默认 Wi-Fi/IP event loop 仍在 core 0，handler 必须短。event post 的返回值、drop 指标和 handler latency 应进入诊断面；WDT user 可用于区分“worker task 活着但内部阶段卡死”。

#### C5 无线并存与低功耗子专题

- **硬件能力证据：** 同一 C5 `soc_caps.h` 声明 Wi-Fi、5 GHz、BLE、IEEE 802.15.4 均支持；这是研究多协议调度的能力前提，不等于协议可无成本同时满载。证据：[C5 capabilities](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/components/soc/esp32c5/include/soc/soc_caps.h)。
- **OpenThread 支持分级：** build rules 对 `ot_cli`/`ot_rcp` 以 `SOC_IEEE802154_SUPPORTED == 1` 启用，所以 C5 属于可构建 target；但自动测试明确只跑 H2/C6。C5 的 light-sleep 示例未被禁用，deep-sleep 示例却因 `TZ-958` 被临时禁用。证据：[build rules](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/openthread/.build-test-rules.yml#L40-L78)。最终文档应写成“CLI/RCP 可构建、测试覆盖有限、light sleep 路径存在、deep sleep 当前禁用”，不能只写“支持 OpenThread”。
- **OpenThread 事件组织：** CLI 示例注册 3 个 eventfd，分别服务 netif、OpenThread task queue、radio driver，再启动默认 ESP event loop 和 OpenThread；网络自动启动前用 OpenThread lock 串行读取 active dataset。证据：[CLI startup](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/openthread/ot_cli/main/esp_ot_cli.c#L48-L93)、[dataset lock](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/openthread/ot_common_components/ot_examples_common/ot_network.c#L21-L27)。
- **PM lock 的阶段性释放：** light-sleep sleepy-device 在启动时创建并持有 `ESP_PM_CPU_FREQ_MAX` lock；只有角色进入 `OT_DEVICE_ROLE_CHILD` 后才 release+delete，让入网阶段保持性能、稳定入网后允许降频/light sleep。证据：[PM lock/state callback](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/openthread/ot_sleepy_device/light_sleep/main/esp_ot_sleepy_device.c#L48-L104)。这比“永远禁用省电”或“启动即省电”更适合状态驱动设备。
- **优先级显式取舍：** 同一示例在调试模式用 FreeRTOS software timer 输出 sleep flags，并明确说明这样使调试工作低于 OpenThread 优先级。证据：[debug timer](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/openthread/ot_sleepy_device/light_sleep/main/esp_ot_sleepy_device.c#L183-L190)。
- **BLE C5 低功耗证据：** NimBLE power-save 示例 target 表和 `sdkconfig.defaults.esp32c5`/`sdkconfig.48m.esp32c5` 明确覆盖 C5，分别演示 32 kHz 与主 XTAL 作为低功耗时钟；README 同时要求 tickless idle 和 power management。证据：[power-save README](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/bluetooth/nimble/power_save/README.md#L1-L34)。其“Hardware Required”文字仍漏列 C5，说明 target 表+配置文件比概括段可靠。
- **BLE host task 与系统 PM 分工：** `app_main` 配置动态频率上下限，并只在 tickless idle 开启时允许 automatic light sleep；随后 `nimble_port_freertos_init()` 启动 host task，host task 长期阻塞在 `nimble_port_run()`，仅在 stop 后 deinit。协议事件调度与系统空闲省电是两层机制，不应由业务 loop 手写轮询。证据：[PM/host task](https://github.com/espressif/esp-idf/blob/f0887bcf8763266effe3fa0b358340df226a04b5/examples/bluetooth/nimble/power_save/main/main.c#L550-L640)。
- **资源预算通过配置裁剪：** 同一示例的 `sdkconfig.defaults.mini` 通过 max connections/bonds/CCCD、角色、GATT client、服务和 transport buffer 数量控制内存，而不是只调低 task stack。这提示 C5 单核/有限内存优化应先裁剪协议能力和并发上限，再做栈尺寸微调。
- **对 ESP-111 的价值：** 若未来 C5 需要省电，应以链路状态机控制 PM lock：连接/校准/高频采集时保性能，稳定/空闲后释放；CSI 当前显式 `WIFI_PS_NONE` 则属于另一种“测量稳定优先”策略，两者不能同时当默认，需要按模式切换并实测 cadence/丢包/功耗。

### `espressif/esp-matter`（深读 3）

- **快照/许可：** `0b82a30549606689e6ea050d7579b382406316dc`，根 Apache-2.0；Light 示例文件多为 CC0/public-domain，Matter 认证/商标与第三方权利不随源码许可自动授予。
- **C5/S3 支持证据分级：** 文档 target、Light CMake board HAL 与两套 target defaults 都包含 C5/S3；C5 还有 Thread-only、Wi-Fi+Thread 预设。可是通用 Light build whitelist 没有 C5/S3，并注明其他 target 尚未测试；现有 S3 pytest 条目实际把 S3 用作 Thread Border Router，不是 S3 Light 闭环。结论只能是“配置/源码支持”，不是当前 CI/硬件通过。
- **任务拓扑：** `app_main` 建数据模型和 HAL 后启动 Matter；框架创建 CHIP event-loop task，可选 C5 OpenThread task。Server 初始化通过 `PlatformMgr().ScheduleWork()` 串行投到 CHIP task，调用者用 FreeRTOS task notification 等待完成；Light 自己没有业务 `QueueHandle_t`。证据：[chip init](https://github.com/espressif/esp-matter/blob/0b82a30549606689e6ea050d7579b382406316dc/components/esp_matter/esp_matter_core.cpp#L287-L352)。
- **事件语义：** commissioning/fabric delegate 用 `PlatformMgr().PostEvent()`，入队失败只 log/drop；应用 event callback 应保持短。`ScheduleWork`、CHIP event queue、OpenThread netif/task queue 是三种不同机制，不能把 OpenThread 的 queue size=10 当作 Matter event queue 容量。
- **本地/远端状态统一：** 远端属性 `PRE_UPDATE` 先驱动硬件，失败可阻止数据模型提交；本地按钮也调用 `attribute::update()`，因此两条路径复用相同回调和报告链。这种“单一状态入口 + HAL 分层”值得借鉴。
- **延迟持久化不是 sliding debounce：** 非易失属性第一次变化时若 timer 未激活才启动 3 秒 timer；后续变化不重置 timer，timer 到点写当前值。它是 fixed-window coalescing。写 NVS 的返回值在 timer 和立即路径均未处理。证据：[deferred write](https://github.com/espressif/esp-matter/blob/0b82a30549606689e6ea050d7579b382406316dc/components/esp_matter/data_model/esp_matter_data_model.cpp#L655-L665)、[schedule](https://github.com/espressif/esp-matter/blob/0b82a30549606689e6ea050d7579b382406316dc/components/esp_matter/data_model/esp_matter_data_model.cpp#L752-L768)。
- **静态启动死等窗口：** `InitializeStaticResourcesBeforeServerInit()` 失败会直接 return，跳过函数末尾唯一 `xTaskNotifyGive()`；调用者在 `xTaskNotifyWait(..., portMAX_DELAY)` 等待。证据：[early return/notify](https://github.com/espressif/esp-matter/blob/0b82a30549606689e6ea050d7579b382406316dc/components/esp_matter/esp_matter_core.cpp#L177-L250)、[wait](https://github.com/espressif/esp-matter/blob/0b82a30549606689e6ea050d7579b382406316dc/components/esp_matter/esp_matter_core.cpp#L316-L352)。这是静态推断、未做故障注入；正确模式应让 completion 携带 status，并保证所有 exit path 都通知。
- **对 ESP-111 的价值：** 协议栈单线程 ownership、worker 投递、统一属性入口和 fixed-window persistence 都可参考；强一致状态不能照搬 log/drop，持久化必须增加失败计数/重试，任何跨任务 startup barrier 都应 bounded wait + result。

### `espressif/esp-adf`（深读 5）

- **快照/许可：** `b875a6e3385f730a49794a59954d4b7940502962`（`release/v2.x`），根为 Espressif MIT：只免费许可在 Espressif 产品上使用，不等同标准 MIT；第三方 codec/组件另计。
- **S3 支持证据：** README 明列 ESP32-S3 与 Korvo-2/BOX 板；仓库有大量 `sdkconfig.defaults.esp32s3` 覆盖 player/recorder/speech/display/OTA/power-save。C5 虽列在框架支持 SoC 中，但成熟音频板/示例配置明显以 S3 为主，不能用 README 一句推断两者同等覆盖。
- **pipeline 不是单任务：** 每个有 stack 的 audio element 创建独立 task；element config 显式携带 `task_stack/task_prio/task_core/out_rb_size/stack_in_ext`。默认是 2 KiB、priority 5、core 0、8 KiB output ringbuffer。证据：[element config/defaults](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/include/audio_element.h#L157-L190)、[task create](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/audio_element.c#L1085-L1119)。S3 双核不会自动分摊 element，必须主动把适合的 decoder/algorithm/I/O element 分核并测量。
- **数据面与控制面分离：** element 之间由 pipeline 创建/复用 ringbuffer；命令和状态由 internal/external queue，listener 用 QueueSet 聚合。element 默认两个 event queue 都只有 5 项，send 是零等待，满时返回失败并 warning；状态必达不能依赖它。证据：[event iface create/read](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/audio_event_iface.c#L51-L155)、[zero-wait send](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/audio_event_iface.c#L241-L277)。
- **ringbuffer 有显式结束语义：** reader/writer 可按 timeout 阻塞；`rb_done_write()` 表示正常 EOF，`rb_abort()` 同时设置 read/write abort 并 give semaphore 唤醒双方。element stop 先 abort 输入/输出 ringbuffer，避免 task 永久卡在数据 I/O。证据：[read/abort/done](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/ringbuf.c#L161-L250)、[abort/done](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/ringbuf.c#L335-L380)、[element stop](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/audio_element.c#L1245-L1269)。
- **task lifecycle：** element task 用 EventGroup 发布 CREATED/STOPPED/RESUMED/DESTROYED 状态，退出前 close、free buffer、记录 stack high-water 并 self-delete。start 等 CREATED 有有限 timeout；resume/stop 也有明确状态位。证据：[task loop/exit](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/audio_element.c#L460-L507)。
- **PSRAM/亲和性 caveat：** `audio_thread_create` 能选 external stack 和 core；未满足外部栈配置时 warning 后退回普通 pinned task。其抽象源自多代 IDF/补丁，复用理念时应优先使用当前 IDF v5.5 原生 caps API，而不是移植 ADF compatibility layer。证据：[audio thread](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_sal/audio_thread.c#L49-L100)。
- **静态清理缺口：** `audio_event_iface_init()` 在 queue 创建中途失败时删除已建 queues，但没有释放最先分配的 `evt` 结构；这是静态可见的小泄漏路径，未做 fault injection。证据：[init failure](https://github.com/espressif/esp-adf/blob/b875a6e3385f730a49794a59954d4b7940502962/components/audio_pipeline/audio_event_iface.c#L65-L99)。
- **对 ESP-111 的价值：** voice/audio 数据应使用专门 ringbuffer/stream contract，并区分 EOF/abort/timeout；控制事件保持小而有界。S3 若分核，要用真实 Wi-Fi/CSI/audio latency profile 选择，而不是照搬 ADF 默认 core 0 或任意绑核。stop 路径必须先唤醒所有阻塞 I/O，再等待 task completion。

- **显式线程池拓扑：** `TaskScheduler` 用 Boost.Asio `io_context` 驱动 immediate/delayed/periodic task，每个 `ThreadConfig` 创建一个可指定 name、core、priority、byte stack 和 external-stack 的 worker；空闲时按 poll interval 睡眠。console 示例明确用 core 0/core 1 两个 10 KiB worker，ServiceManager 的 Kconfig 默认则是 3 个 priority 5、15 KiB、external-stack worker，core 为 0、1、no-affinity。证据：[worker creation](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/utils/brookesia_lib_utils/src/task_scheduler.cpp#L35-L124)、[console topology](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/examples/service/console/main/main.cpp#L66-L116)、[ServiceManager worker config](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/service/framework/brookesia_service_manager/Kconfig#L35-L109)。
- **串行组与周期语义：** group 可绑定 Asio strand，使同组 one-shot 串行执行；periodic callback 返回 `false` 才结束。下一次 timer 是在本次 callback 返回后才重新 `expires_after(interval)`，并额外用原子 `is_executing` 拒绝重入，因此实际是 fixed-delay、同 task 不重叠，而不是固定节拍补偿或积压队列。长 callback 会直接拉长实际周期。证据：[group contract](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/utils/brookesia_lib_utils/include/brookesia/lib_utils/task_scheduler.hpp#L166-L190)、[periodic implementation](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/utils/brookesia_lib_utils/src/task_scheduler.cpp#L974-L1048)。
- **取消、等待和停止合同：** scheduler 保存 promise/future，可对单 task/group/all 做有限等待；停止时把待办标为 canceled、完成 promise、停止 `io_context`、interrupt 并 join 所有 worker，再清空 task/group/strand。好处是 teardown 路径明确；限制是 `join_all()` 本身没有 timeout，若运行中的 callback 不经过可中断点或永久阻塞，`stop()` 仍可能长期卡住，这是静态可见风险、未做故障注入。证据：[bounded waits](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/utils/brookesia_lib_utils/src/task_scheduler.cpp#L557-L655)、[stop](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/utils/brookesia_lib_utils/src/task_scheduler.cpp#L127-L182)。
- **观测面是调度设计的一部分：** scheduler 原子统计 total/completed/failed/canceled/suspended；`ThreadProfiler` 通过同一 scheduler 周期投递“两次 FreeRTOS task snapshot + 间隔计算”，输出 core、stack high-water、state、CPU 百分比和阈值 signal，停止等待有 100 ms 上限。它要求启用 FreeRTOS runtime-stats 配置，且 profile 本身也消耗 worker 与内存，适合诊断模式而非默认高频常开。证据：[scheduler statistics](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/utils/brookesia_lib_utils/src/task_scheduler.cpp#L771-L790)、[profiler scheduling/stop](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/utils/brookesia_lib_utils/src/thread_profiler.cpp#L72-L205)、[sampling](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/utils/brookesia_lib_utils/src/thread_profiler.cpp#L251-L268)。
- **ServiceManager 的 ownership 模型：** 服务依赖先做拓扑排序；`ServiceBinding` 是 move-only RAII handle，bind 时递归持有 dependency bindings、引用计数从 0 到 1 才 start，析构时先 unbind 自身再逆序释放依赖。服务函数可直接执行或按 schema 投递到 service task group，同步包装器用 future 和默认 500 ms 有界等待；结果 handler 在释放 service state lock 后调用，源码明确用于避免 re-entry deadlock。证据：[binding model](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/service/framework/brookesia_service_manager/include/brookesia/service_manager/service/manager.hpp#L27-L120)、[bind/start](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/service/framework/brookesia_service_manager/src/service/manager.cpp#L261-L390)、[function dispatch](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/service/framework/brookesia_service_manager/src/service/base.cpp#L48-L139)。
- **一个并发生命周期风险：** `unbind()` 在锁内把最后引用减到 0 并立刻把 state 设为 `Idle`，随后解锁才执行 `service->stop()`；状态机没有 `Stopping`。因此另一线程可能在旧 stop 尚未完成时看到 `Idle` 并重新 start 同一服务。源码注释声称避免 bind/stop 干扰，但当前状态顺序没有完全封住这个窗口；这是静态推断，未运行复现。证据：[unbind window](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/service/framework/brookesia_service_manager/src/service/manager.cpp#L393-L444)、[state enum](https://github.com/espressif/esp-brookesia/blob/39871f5f0f491fec476e54b8736661e73f3c8997/service/framework/brookesia_service_manager/include/brookesia/service_manager/service/manager.hpp#L274-L291)。
- **对 ESP-111 的价值：** 可借鉴“线程参数集中配置、串行 ownership group、有限 wait、RAII dependency、调度统计+task profile”这组完整合同。不可直接移植 Boost/C++23 服务层到现有固件，也不能把 fixed-delay scheduler 当成 CSI 精确采样时钟；S3 若引入 service lifecycle，必须先补 `Stopping`/generation 或单线程 owner，并给 stop callback 设协作取消和总体 deadline。

- **模型 ownership：** `esp_srmodel_init()` mmap model partition 并返回进程级单例；refcount 是无同步全局整数，普通 deinit 不递减，refcount deinit 在最终释放后也不归零。安全顺序是停止并 join feed/fetch -> destroy AFE/识别实例 -> deinit model mapping，由单一 manager 管理且不混用两种 deinit。证据：[model init/deinit](https://github.com/espressif/esp-sr/blob/7ff63a7da40e15e502681be48c4d0e78475544a3/src/model_path.c#L308-L377)、[refcount paths](https://github.com/espressif/esp-sr/blob/7ff63a7da40e15e502681be48c4d0e78475544a3/src/model_path.c#L510-L554)。
- **示例停止反例：** 文档 feed task 无限循环，fetch task 遇错却直接 destroy 两者共享的 `afe_data`，没有先停 feed/join，存在 use-after-free 风险；API 本身也没有 stop，只有 reset/destroy。产品必须由 supervisor 执行 `RUNNING -> STOP_REQUESTED -> feed stop -> fetch bounded exit -> both joined -> destroy -> model deinit`。证据：[document example](https://github.com/espressif/esp-sr/blob/7ff63a7da40e15e502681be48c4d0e78475544a3/docs/en/audio_front_end/README.rst#L158-L200)、[lifecycle API](https://github.com/espressif/esp-sr/blob/7ff63a7da40e15e502681be48c4d0e78475544a3/include/esp32s3/esp_afe_sr_iface.h#L195-L235)。
- **对 ESP-111 的价值：** 音频子系统只向外发布带 owner 的有界消息，禁止把 AFE borrowed pointer 过队列；feed cadence 来自 I2S/DMA 与查询到的 chunk size，不用普通 delay 模拟；ring pressure、fetch timeout、DMA overrun、drop/late frame 分开计量。上述结论经固定链接复核，但本轮未运行声学、过载或 CI 实验。

- **对 ESP-111 的价值：** camera/CSI/telemetry 的“状态类流”都可采用 latest-only + timestamp，但任何外部 buffer 必须加 lease/refcount 或复制，不能只靠 ring 深度猜生命周期。若做过载恢复，应把丢帧/清状态明确为策略并计数，不能用最高优先级监控 task 掩盖长期 CPU 预算错误。

- **典型单 owner task：** on/off coordinator 创建一个 `Zigbee_main`（4 KiB、priority 5、no affinity），依次 init、注册 device/endpoint/cluster、start、阻塞 launch mainloop，返回后 deinit/self-delete。C5 单核必须把它与 Wi-Fi/CSI/timer/GPIO task 联合 profile。证据：[main loop contract](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/components/esp-zigbee-lib/include/esp_zigbee.h#L126-L170)、[task](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/home_automation_devices/on_off_light/main/on_off_light.c#L166-L190)。
- **跨任务必须 lock/post：** API 默认不线程安全；main-loop callback 可直接调用，其他 task 要取得 Zigbee lock 或 `esp_zigbee_task_queue_post()`。公开 API 不给 queue depth、满载策略/公平性，v2 还移除旧 queue-size setter，因此 post failure 必须作为真实背压，不可假设必达。证据：[API contract](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/components/esp-zigbee-lib/include/esp_zigbee.h#L172-L209)、[thread safety guide](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/docs/en/developing.rst#L361-L392)。
- **commissioning 重试过于简单：** coordinator 状态为 initialization -> formation -> steering；非 factory-new 从 NVS 恢复并开放网络。失败统一安排固定 1 秒 retry，无 reason 分类/backoff/jitter/上限。v2 示例用 heap one-shot `esp_timer` 替代已删除的 scheduler alarm；helper清理自身资源，但调用方忽略 schedule 失败且没有 cancel handle/generation 去重，重复 signal 可能挂多个重试。证据：[signal flow](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/home_automation_devices/on_off_light/main/on_off_light.c#L41-L105)、[alarm helper](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/utils/alarm_timer/src/alarm_timer.c#L19-L68)。
- **descriptor ownership 不清：** device/endpoint/cluster 是 opaque pointer，create/free API存在，但 register/link 是否转移 ownership未写清；示例不检查 invalid handle、不保留/释放，中途失败由 `ESP_ERROR_CHECK` 终止。表容量可配置但默认值/满表策略在闭源栈内。证据：[descriptor APIs](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/components/esp-zigbee-lib/include/ezbee/af.h#L141-L376)、[example registration](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/home_automation_devices/on_off_light/main/on_off_light.c#L138-L153)。
- **NVS/deep-sleep 边界：** v2 Zigbee datasets迁到独立 16 KiB `zb_storage` NVS；示例没有 no-free-pages/new-version erase-reinit。deep-sleep ZED 在异步 match/bind 后只等固定 5 秒就睡，没有等 callback/cancel；pytest 验证 sleep/rejoin，不验证 binding 完成，拥塞下存在被睡眠截断的待测窗口。证据：[partitions/init](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/home_automation_devices/on_off_light/partitions.csv#L1-L8)、[sleep window](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/sleepy_devices/deep_sleep_end_device/main/deep_sleep_end_device.c#L114-L203)、[pytest boundary](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/pytest_esp_zigbee_open_examples.py#L136-L154)。
- **文档/实现冲突提示审计方法：** switch header 声称 callback 在 ISR，实际 ISR 只入 depth-10 queue，priority-10 `button_detected` task才调用 callback；实现还忽略 task create、ISR install失败遗留 task/queue、deinit不清理。这说明应以实现为准，并把示例视作启动一次活到重启。证据：[header](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/utils/switch_driver/include/switch_driver.h#L38-L66)、[implementation](https://github.com/espressif/esp-zigbee-sdk/blob/7eff0fbe19bcf2acd112ba0b5f080530efc49626/examples/utils/switch_driver/src/switch_driver.c#L23-L116)。
- **对 ESP-111 的价值：** 保留单 stack-owner task，非协议 callback 只投 typed event/有限 lock；commissioning/leave/factory-reset/bind result是控制流，必须确认投递和重试。补齐 NVS version/watermark/recovery、generation 去重、bounded backoff和深睡前 pending-operation barrier，并在 C5 单核上实测 Wi-Fi/CSI/802.15.4 共存。

## ESP-111 映射

当前映射基于 2026-07-10 live worktree，完整证据见 `agent-esp111-mapping.md`；未构建、未上板、未做负载测试。

| 外部模式 | ESP-111 当前实现 | 判断 |
|----------|-----------------|------|
| 默认 `sys_evt` + 专用业务 worker | C5 除隐藏 `sys_evt` 外另有 dispatcher + CSI/BME/system workers | 直接保留；拓扑图必须计入系统 task |
| event data copy / pointer 失败释放 | C5 event 按值复制；S3 owned event/payload 在 push/替换/丢弃时统一 release | 直接保留 ownership 规则 |
| producer 零等待 + 检查 queue-full | C5 零等待并聚合 drop/depth/latency 诊断 | 直接保留；critical 另定义有限重试 |
| task/user 两级 WDT | S3 有 task-level WDT；C5 dispatcher/workers 未见订阅 | 先实验 C5 task WDT，再评估 CSI report/HTTP named user |
| C5 单核、S3 双核且 `sys_evt` 在 core 0 | C5 普通 task 正确地共享唯一核；S3 scheduler/protocol/CSI/stream 也都未绑核 | S3 只在 profile 后实验把 CPU-heavy worker 放 core 1 |
| CSI callback 复制后交 worker | C5 callback 不 malloc/打印/HTTP，只覆写最多 64 I/Q pair 的 latest slot并发 `CSI_READY` | 本地实现比 callback malloc demo 更稳，直接保留 |
| 网络 handler 必须短且幂等 | S3 handler 主要更新小状态并投递到预创建 network worker | 禁止照搬 got-IP 时重复创建 producer task |
| caps task stack 可放 PSRAM但要配对 delete | S3 UDP ingress task stack 已放 PSRAM，当前生命周期活到重启 | 先测 latency/drop/WDT/high-water；未来删除必须用 caps API |

最明确的禁止项是：CSI callback 整帧打印、pointer queue 投递失败不释放、重复网络事件里无幂等创建 task，以及把 `portMAX_DELAY` 用在高频 telemetry producer。
