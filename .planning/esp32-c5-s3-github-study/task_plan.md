## 已做决策

| 决策 | 理由 |
|------|------|
| 使用独立 `.planning/` 作用域 | 根规划文件混有多轮旧任务，避免长期上下文互相污染 |
| 最终输出采用多章手册目录 | 项目广度、源码深度和横向比较需要不同的信息密度 |
| 以 shallow clone/API/固定 permalink 为主要证据 | README 概览不足以支持代码与调度结论 |
| 官方项目打基础、社区项目验证工程取舍 | 同时覆盖能力边界与真实产品化模式 |

## 遇到的错误

| 错误 | 尝试次数 | 解决方案 |
|------|---------|---------|
| 新建 goal 时系统报告已有未完成目标 | 1 | `get_goal` 确认该目标就是用户本轮原文，直接续接 |
| 首次并行读取技能与记忆输出被截断 | 1 | 按文件和行段重新读取，后续控制单次输出规模 |
| 首轮 Web 四查询返回流解码失败 | 1 | 改用直接打开官方仓库 + GitHub API/浅克隆，不原样重试 |
| 压缩后网络沙箱阻止 sparse checkout 补拉 OpenThread component blob | 1 | 使用已下载 build rules/示例作分级证据，不声称未读内部实现；后续优先现有快照 |
| 7 个 partial clone 的首次 sparse 展开均被网络沙箱拒绝 | 1 | 单独批准只读 `git sparse-checkout` 前缀后成功展开；未执行仓库代码 |
| 补记 BLE 证据时 patch 锚点放错目标文件 | 1 | 确认补丁未部分应用，改用按文件拆分的小补丁 |
| 批次 2D 进度日志误用了预计结束时间 | 1 | 读取实际时钟后改正，后续批次结束再记录时间 |
